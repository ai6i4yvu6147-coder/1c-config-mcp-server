## Протокол тестирования

Цель этого протокола — исключить «фиктивные» проверки (например, создание технических БД агентом) и проверять функциональность на **реальном (подключённом в IDE) MCP**.

### Определение: «реальный MCP»

Под **реальным MCP** здесь понимается сервер, который **уже подключён в IDE** (Cursor/другая) и вызывается как MCP-клиентом.

- Название подключения в IDE может отличаться от названия проекта/исполняемого файла.
- Для однозначности всегда начинайте проверку с `active_databases`: по его выводу видно, что вы подключены к нужной рабочей среде.

### Общий порядок тестирования (справочно)

1. **Статические проверки** проекта (если применимо): линтер/форматтер/прочие проверки, которые не требуют пересоздания БД.
2. После успешных правок пользователь **самостоятельно пересобирает сервер** и обновляет его в рабочей папке.
3. Пользователь **переподключает MCP** в агенте (перезапуск/переподключение клиента).
4. Фактическая проверка работы выполняется **вызовами MCP tools** на рабочем MCP (например: `active_databases`, `find_object`, `search_code`, и т.д.).

### Запрещённые действия для агента

Агенту нельзя:

- имитировать парсинг выгрузки 1С «для теста»;
- создавать/заводить «технические» SQLite базы ради проверки (и смотреть их содержимое);
- писать/предлагать миграции или конвертации существующих БД (см. `docs/database.md`: **NO_DB_MIGRATIONS**);
- подменять проверку MCP tools прямыми запросами к SQLite «для доказательства работоспособности».
