## Протокол тестирования

Цель этого протокола — исключить «фиктивные» проверки (например, создание технических БД агентом) и проверять функциональность на **реальном (подключённом в IDE) MCP**.

### Канонический рабочий цикл (кто что делает)

1. Пользователь ставит задачу агенту (изменения в исходниках).
2. Агент делает правки в исходниках и завершает работу.
3. Пользователь **сам** пересобирает portable/server, **сам** пересобирает базы через Admin Tool и **сам** переподключает MCP в IDE.
4. Агент выполняет проверку **только через вызовы MCP tools** на «реальном MCP» (начиная с `active_databases`).

### Определение: «реальный MCP»

Под **реальным MCP** здесь понимается сервер, который **уже подключён в IDE** (Cursor/другая) и вызывается как MCP-клиентом.

- Название подключения в IDE может отличаться от названия проекта/исполняемого файла.
- Для однозначности всегда начинайте проверку с `active_databases`: по его выводу видно, что вы подключены к нужной рабочей среде.

### Общий порядок тестирования (справочно)

1. **Статические проверки** проекта (если применимо): линтер/форматтер/прочие проверки, которые не требуют пересоздания БД.
2. После успешных правок пользователь **самостоятельно пересобирает сервер** и обновляет его в рабочей папке.
3. Пользователь **переподключает MCP** в агенте (перезапуск/переподключение клиента).
4. Фактическая проверка работы выполняется **вызовами MCP tools** на рабочем MCP (например: `active_databases`, `find_object`, `search_code`, и т.д.).

### Запрещённые действия для агента

Агенту нельзя:

- имитировать парсинг выгрузки 1С «для теста»;
- создавать/заводить «технические» SQLite базы ради проверки (и смотреть их содержимое);
- писать/предлагать миграции или конвертации существующих БД (см. `docs/database.md`: **NO_DB_MIGRATIONS**);
- подменять проверку MCP tools прямыми запросами к SQLite «для доказательства работоспособности».

### Важно: выгрузка 1С — источник истины, но это не «тестирование MCP»

Если задача касается парсера (`shared/xml_parser.py`) или импорта в БД (`admin_tool/db_manager.py`), то **источник истины** — реальные файлы выгрузки 1С (XML/BSL).

- Если пользователь не дал путь к выгрузке, агент должен **запросить путь** (вне репозитория; не копировать выгрузку в проект).
- Агент может смотреть конкретные файлы выгрузки, чтобы корректно реализовать парсинг.
- Но итоговую проверку функциональности всё равно делаем **на реальном MCP** (через вызовы tools), а не «локальным прогоном парсера».

### Важно: не подменять «реальный MCP» локальным запуском из репозитория

Иногда кажется удобным «быстро проверить», запустив Python-код из репозитория (например, создать `ConfigurationTools(...)`
и дернуть методы напрямую). Это **не считается** проверкой по этому протоколу, потому что:

- вы проверяете другой набор `projects.json`/`databases/`, который может не совпадать с рабочим окружением IDE;
- вы обходите сам MCP-контур (transport/serialization/валидации и т.д.);
- это приводит к ложным выводам вроде «нет проектов/баз», хотя в рабочем MCP всё есть.

Правильная проверка: **только вызовами MCP tools через IDE-подключение**, начиная с `active_databases`.
