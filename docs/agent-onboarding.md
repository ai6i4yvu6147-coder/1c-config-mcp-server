## Agent onboarding (контекст для ИИ)

### Коротко о проекте

Проект индексирует выгрузку конфигурации 1С (XML + BSL) в SQLite и предоставляет MCP-инструменты для поиска/анализа кода и метаданных.

### Ключевые политики проекта (не нарушать)

- **NO_DB_MIGRATIONS**: никогда не писать миграции/конвертации существующих SQLite-баз в `databases/`. После изменений схемы/логики импорта базы **всегда пересоздаются** через `admin_tool` из исходной выгрузки.
- **`INDEXER_VERSION`**: при несовместимых изменениях схемы/данных в БД увеличивать константу в `shared/indexer_version.py` (см. `.cursor/rules/bump-indexer-version.md`, подробности в `database.md`).
- **Whitelist типов метаданных**: парсер намеренно обрабатывает только типы из `shared/xml_parser.py` (`object_types`). Новые типы добавляются **точечно** по мере необходимости, с учётом специфики конкретного типа.
- **Тестирование**: функциональная проверка делается на «рабочем» MCP после пересборки сервера пользователем и переподключения MCP в агенте. Проверка — **только через вызов MCP tools**, без имитации парсинга и без «технических» БД.

### Термины (чтобы не было путаницы)

- **«Реальный MCP» / «рабочий MCP»**: MCP-сервер, который **уже подключён в IDE** (Cursor/другая), и проверка делается через **фактические вызовы его tools**.
- **Важно про имена**: имя подключения в IDE может не совпадать с названием проекта/папки/экзешника.
  - В Cursor и MCP-описаниях есть как минимум 2 идентификатора: `serverIdentifier` и `serverName`.
  - Поэтому для однозначности ориентируемся не на название, а на наличие нужных tools (`active_databases`, `list_objects`, `search_code`, …) и на фактический результат `active_databases`.
