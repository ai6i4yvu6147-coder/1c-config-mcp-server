# ТЗ: Модификация MCP сервера 1C-mcp-configuration

## 1. Проблема 1 — `search_code` возвращает неполные результаты по модулю

### Описание
Параметр `max_results` работает как глобальный лимит на всё множество найденных совпадений, а не как лимит на один модуль. При наличии нескольких совпадений в одном модуле и нескольких модулей в результатах — часть совпадений внутри модуля молча отбрасывается без каких-либо предупреждений.

### Пример
Поиск `ФТ_НоменклатурнаяГруппа` с `max_results=50` в расширении вернул из `СписаниеСРасчетногоСчета.ManagerModule` только 1 результат, хотя фактически там было как минимум 2 метода с вхождениями.

### Ожидаемое поведение
Вариант А (предпочтительный): `max_results` трактуется как лимит **на один модуль**, а не глобально. Т.е. из каждого модуля возвращается до `max_results` совпадений.

Вариант Б: если по модулю найдено больше совпадений, чем попало в результат — добавлять в вывод явное предупреждение вида:
```
⚠ показано 1 из 3 совпадений в этом модуле, увеличьте max_results или уточните запрос
```

## 2. Проблема 2 — несогласованность `search_code` и `get_procedure_code`

### Описание
`search_code` находит функцию по строке совпадения, но при попытке получить её код через `get_procedure_code` с тем же именем возвращается ошибка "процедура не найдена". Инструменты используют разные механизмы поиска/индексации и их результаты не согласованы.

### Пример
`search_code` нашёл функцию `ДополнитьЗапросРеквизитами` в `CommonModule.ФТ_ДополнительныеРеквизиты.Module`. Вызов `get_procedure_code` с `object_name="ФТ_ДополнительныеРеквизиты"` и `procedure_name="ДополнитьЗапросРеквизитами"` вернул ошибку.

### Причина (предположение)
`get_procedure_code` ищет процедуры в объектных модулях (`ObjectModule`, `ManagerModule`, `FormModule`) и по умолчанию использует `module_type="Module"` для обычных модулей. Возможно, для `CommonModule` логика поиска отличается или имя объекта воспринимается иначе.

### Ожидаемое поведение
Любая процедура/функция, обнаруженная через `search_code`, должна быть доступна через `get_procedure_code` с теми же идентификаторами (object_name, module_type), которые `search_code` возвращает в заголовке результата. Если это технически затруднено — `search_code` должен явно указывать `module_type` в заголовке каждого результата, чтобы вызов `get_procedure_code` можно было сформировать корректно.

## 3. Проблема 3 — сниппеты `search_code` не указывают имя метода

### Описание
Результаты `search_code` показывают фрагмент кода вокруг совпадения, но не указывают, в какой именно процедуре или функции это совпадение находится. При большом модуле определить принадлежность совпадения к конкретному методу можно только визуально по контексту — это ненадёжно и трудоёмко.

### Ожидаемое поведение
В заголовке каждого результата добавить поле `Процедура/Функция`. Пример формата:

```
• CommonModule.УчетДенежныхСредств.Module
  Функция: ФТ_СформироватьДвиженияПрочееСписание
  ...фрагмент кода...
```

Если совпадение находится вне какой-либо процедуры (например, в теле модуля) — указывать `<тело модуля>`.
